# ChatHub 3.0 UI 视觉方案

**方案名**: Zero Chrome Workspace  
**阶段**: 静态设计基线 V1  
**日期**: 2026-03-25  
**适用范围**: Chrome 扩展主工作台（不含网站 iframe 内部页面）

---

## 1. 设计目标

本方案用于替代当前“常驻侧边栏 + 顶部信息栏 + 底部输入栏”的多层级界面结构，重新建立一套**内容优先、控件退后、空间最大化**的工作台布局。

设计目标如下：

1. AI 聊天窗口必须成为页面唯一主角。
2. 页面横向和纵向空间优先分配给 iframe 聊天内容。
3. 全局控制必须收敛，不得分散在顶部、左侧、右下角多个区域抢占注意力。
4. 界面整体应体现专业工具感，而不是展示型 dashboard。
5. 所有高级感都建立在克制、秩序、精度之上，而不是大面积装饰。

---

## 2. 核心原则

### 2.1 Zero Chrome

默认状态下，页面只保留两类常驻元素：

- 聊天窗口工作区
- 底部统一控制条

以下元素不再常驻占位：

- 顶部全局状态栏
- 左侧模型栏
- 右下角独立浮动入口

### 2.2 Content First

任何装饰性视觉都不得压缩聊天内容区。  
视觉层级顺序固定为：

1. iframe 聊天内容
2. 窗口头部与状态
3. 底部控制条
4. 临时抽屉 / 浮层 / 模态

### 2.3 Progressive Disclosure

高级功能不消失，但默认不抢眼。

- 模型管理通过左侧抽屉出现
- 设置通过抽屉底部入口或底部次级入口进入
- 提示词库通过底部次级入口呼出
- 状态信息合并到底部栏，避免重复展示

- 其余窗口降低亮度
- 不使用重 blur，避免文本变脏
- 底部控制条保持可用

## 5. 视觉风格

### 5.1 关键词

- Quiet
- Precise
- Dense
- Professional
- Unobtrusive

### 5.2 背景

背景不做大面积戏剧化光效，只保留一层非常轻的深色渐变。

建议：

- 主背景：`#07111C -> #04070D`
- 辅助微光：顶部左侧极弱冷蓝晕染，透明度低于 10%
- 页面纹理：可选极弱噪点，不得影响清晰度

### 5.3 色彩层级

| 用途 | 建议色 |
|------|--------|
| 主背景 | `#050A12` |
| 次级面板 | `rgba(11, 18, 30, 0.86)` |
| 输入栏表面 | `rgba(10, 16, 28, 0.92)` |
| 主描边 | `rgba(255, 255, 255, 0.08)` |
| 次描边 | `rgba(255, 255, 255, 0.05)` |
| 文字主色 | `#F3F7FB` |
| 文字次色 | `#93A4B8` |
| 文字弱化 | `#627186` |
| 主强调蓝 | `#4C8DFF` |
| Hover 蓝 | `#69A3FF` |
| 成功绿 | `#35D07F` |
| 准备黄 | `#F0C34B` |
| 异常红 | `#FF6B7A` |

### 5.4 阴影

整体阴影要浅，避免浮夸。

| 用途 | 建议 |
|------|------|
| 窗口基础阴影 | `0 10px 24px rgba(0, 0, 0, 0.22)` |
| 控制条阴影 | `0 12px 28px rgba(0, 0, 0, 0.26)` |
| 抽屉阴影 | `0 18px 48px rgba(0, 0, 0, 0.34)` |
| 焦点高光 | 仅边框蓝色轻微增强，不加大光晕 |

### 5.5 模糊

backdrop-blur 只保留很小剂量，用于表达材质，而非制造“玻璃秀”。

- 底部控制条：`blur(12px)` 以内
- 模型抽屉：`blur(10px)` 以内
- 不建议对窗口主体使用 blur

## 6. 字体与排版

### 6.1 字体策略

建议使用中性、现代、工具感较强的无衬线组合，不用过于品牌化的展示字体。

- 中文优先：`PingFang SC`, `MiSans`, `Noto Sans SC`
- 英文优先：`SF Pro Display`, `Inter`, `Geist`

### 6.2 字号层级

| 层级 | 用途 | 字号 |
|------|------|------|
| L1 | 窗口标题 | 14px / 600 |
| L2 | 输入框正文 | 15px / 400 |
| L3 | 操作按钮 | 13px / 500 |
| L4 | 状态标签 | 11px / 600 |
| L5 | 次要说明 | 12px / 400 |

### 6.3 字距原则

- 大写英文标签才允许轻微 tracking
- 常规中文文本不额外加字距
- 避免大标题，避免营销型文案口气

## 7. 组件设计

## 7.1 主工作区

主工作区承担页面 90% 以上视觉比重。

规则：

- 不再显示顶部栏
- 不再显示常驻左侧栏
- 空状态也要保持克制，不做巨型说明面板
- 空状态建议只保留一句简洁引导和一个入口动作

空状态文案建议：

> 选择模型开始并行对话

辅助文案建议：

> 聊天窗口会出现在这里，便于多模型同时比较输出。

## 7.2 聊天窗口

### 头部结构

```text
[拖拽点]  OpenAI           ● 已就绪                 ↻   □   ×
```

规则：

- 图标块可以保留，但要缩小到次要层
- host 域名默认不展示或仅弱化显示
- 状态标签改为更短、更干净的点状状态
- 双击最大化保留

### 头部视觉建议

- 高度：40px
- 背景：深色、低对比
- 下边界：1px 分割线
- 图标按钮：32px 命中区，视觉尺寸 16px

### 状态表达

| 状态 | 视觉 |
|------|------|
| 已就绪 | 绿色点 + “已就绪” |
| 准备中 | 黄色点 + “准备中” |
| 发送中 | 蓝色点 + “发送中” |
| 异常 | 红色点 + “异常” |
| 不支持 | 灰点 + “不支持” |

状态要足够克制，不能像徽章展示墙。

## 7.3 底部统一控制条

这是唯一常驻的全局控制面板。

推荐布局：

```text
[模型] [Sync] [提示词] [📎] [ 输入框................................ ] [2 窗口] [新对话] [发送]
```

设计要求：

- 单层结构，避免当前偏“大卡片套小卡片”的感觉
- 输入框占最大宽度
- 左右按钮统一高度
- “窗口数”作为信息项弱化显示
- 有附件时再展开文件预览层

### 左侧区

- `模型`：打开左侧抽屉
- `Sync`：广播/草稿切换
- `提示词`：打开提示词库
- `附件`：上传文件

### 中间区

- 输入框作为主视觉
- 默认单行
- 超过一行时向上增长
- 背景对比度略高于控制条底座

### 右侧区

- 窗口数：只做信息标签
- 新对话：次级按钮
- 发送：主按钮

### 输入占位文案

同步模式：

> 把消息发往所有窗口…

草稿模式：

> 先写草稿，按 Enter 注入到当前窗口…

## 7.4 模型抽屉

模型抽屉承担原侧边栏的全部职责，但不再常驻。

内容结构建议：

1. 顶部标题区
2. 已激活模型
3. 全部模型
4. 自定义入口
5. 底部设置入口

### 顶部标题区

仅保留：

- 标题：模型
- 关闭按钮
- 可选副标题：已激活 3 个

### 列表项

每一项建议包含：

- 缩写 / 小图标
- 模型名
- 辅助域名或类型
- 激活状态

不建议：

- 大面积渐变
- 过宽卡片
- 显眼的收藏区块装饰

### 列表项状态

- 未激活：普通文本
- 已激活：轻描边或左侧细条标记
- hover：轻微提亮

## 7.5 提示词库

提示词库不再以右下角单独浮球存在。  
建议改为底部控制条的次级入口。

打开方式：

- 点击底栏“提示词”
- 从右侧或底部弹出轻量面板

要求：

- 不遮挡主要聊天窗口中部
- 不与模型抽屉同时抢占视线
- 优先做成“临时取用”

## 7.6 设置面板

设置不再作为日常高频入口展示。  
建议入口放在模型抽屉底部。

原因：

- 设置属于低频管理行为
- 不应该在主工作台中占视觉权重

## 8. 动效规范

### 8.1 原则

- 快
- 短
- 稳
- 只服务于理解界面状态变化

### 8.2 建议时长

| 动作 | 时长 |
|------|------|
| 按钮 hover | 120ms |
| 抽屉展开 / 收起 | 180ms |
| 底栏焦点增强 | 140ms |
| 聚焦模式切换 | 180ms |
| 状态点变化 | 120ms |

### 8.3 缓动

建议统一使用：

`cubic-bezier(0.22, 0.61, 0.36, 1)`

### 8.4 禁止项

- 大范围弹跳
- 过强缩放
- 夸张发光动画
- 长时间拖尾

## 9. 响应式策略

### 9.1 桌面优先

该扩展以桌面端为主。

建议断点：

- `>= 1600px`: 双列或三列舒展布局
- `1280px - 1599px`: 双列标准布局
- `1024px - 1279px`: 双列紧凑布局
- `< 1024px`: 优先保证窗口可读性，减少边距和控制条占宽

### 9.2 小屏策略

- 底栏按钮从“图标 + 文案”退化为纯图标
- 窗口头部进一步压缩
- 抽屉宽度可降至 `264px`

3. 底部输入栏改为唯一常驻全局控制层
4. 提示词入口从右下角浮球并入底栏
5. 设置入口并入模型抽屉底部

## 11. 非目标

本阶段不追求以下内容：

- 重品牌化视觉包装
- 夸张玻璃拟态
- 强光效和重阴影
- 复杂仪表盘式信息展示
- 对 iframe 内部网页进行统一美化

---

## 12. 结论

这版方案的本质不是“重新画一个更酷的界面”，而是把 ChatHub 从一个“有很多控制层的聚合器”，重塑成一个**让 AI 对话本身占据绝对中心的工作台**。

如果后续按此方案实施，判断是否成功的标准不是“界面是否更炫”，而是：

1. 用户第一眼是否直接看到多个 AI 的输出内容。
2. 用户是否几乎感觉不到全局 UI 对空间的争夺。
3. 所有高频操作是否都能在底部一条控制带内完成。
4. 模型管理是否在需要时快速可达，不需要时完全隐身。
